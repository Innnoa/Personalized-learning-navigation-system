# 教师端学情总览轻量增强设计

日期：2026-06-21

## 1. 设计背景

当前教师课程详情下的“学情总览”页主要承载课程建设视角的信息：

- 课程基础信息
- 课程规模指标
- 内容覆盖缺口
- 章节分布

这套信息对“建课维护”有效，但对教师进入页面后最常见的两个判断帮助不足：

1. 班级整体学情现在处于什么状态
2. 当前应该优先跟进哪些学生

本次目标不是把页面重做成重型驾驶舱，而是在现有页面上做一次轻量增强，让该页同时具备：

- `班级整体判断`
- `重点学生识别`
- `课程建设信息保留`

## 2. 设计目标

本次改造需要满足以下目标：

1. 在现有课程概览页中补充“班级学情总览”信息
2. 同时展示班级整体态势与重点关注学生
3. 优先复用现有前后端接口，不新增后端聚合接口
4. 保持页面仍然以“轻量概览”而非“复杂控制台”为定位
5. 学情区加载失败时，不影响原有课程建设信息展示

## 3. 非目标

本次明确不做：

- 不新增后端接口
- 不新增趋势图、时间序列图或复杂筛选器
- 不把学生详情能力内嵌到本页
- 不替代现有“学生列表”页和“学生详情”页
- 不移除现有课程建设信息模块

## 4. 数据复用方案

本次页面继续复用现有三个接口。

### 4.1 课程建设数据

接口：

`GET /api/teacher/courses/:courseCode/overview`

用途：

- 课程基础信息
- 内容覆盖率
- 未覆盖知识点
- 章节分布

### 4.2 班级整体学情

接口：

`GET /api/teacher/courses/:courseCode/stats`

已提供字段：

- `studentCount`
- `avgMastery`
- `masteryDistribution.low.count`
- `masteryDistribution.mid.count`
- `masteryDistribution.high.count`

用途：

- 学生总数
- 平均掌握度
- 掌握度分层

### 4.3 学生个体风险线索

接口：

`GET /api/teacher/courses/:courseCode/students`

已提供字段：

- `learnerCode`
- `learnerName`
- `major`
- `gradeLabel`
- `overallMastery`
- `lastActivityAt`

用途：

- 生成重点关注学生名单
- 提供跳转到学生详情页的入口

## 5. 页面结构设计

页面仍保留当前“课程概览”结构，在课程基础信息和课程指标卡之后插入新的学情模块。

推荐结构如下：

1. 课程基础信息卡
2. 课程指标卡
3. `班级学情总览`
4. `内容覆盖 / 覆盖缺口 / 维护建议`
5. 章节分布

这样页面形成两条并列主线：

- 上半区先看“学生学得怎么样”
- 下半区继续看“课程内容建设得怎么样”

## 6. 学情模块设计

学情模块由两个子块组成。

### 6.1 班级态势

展示形式：3 到 5 张轻量统计卡 + 1 个分层条形区

建议内容：

- 学生总数
- 平均掌握度
- 高掌握人数（70% - 100%）
- 中等掌握人数（35% - 70%）
- 低掌握人数（0% - 35%）

其中：

- `学生总数` 和 `平均掌握度` 直接作为主指标
- `高 / 中 / 低` 分层继续使用当前已有统计接口中的分布信息
- 展示方式保持纯 CSS 卡片和横向分布条，不引入新图表库

页面文案上应强调这是一眼判断班级状态的区域，而不是细粒度分析面板。

### 6.2 重点关注学生

展示形式：列表卡片，默认展示前 5 名

每名学生展示以下字段：

- 学生姓名
- 专业 + 年级
- 综合掌握度
- 最近活动时间
- `查看详情` 入口

如果 `lastActivityAt` 缺失，应明确展示为“暂无最近活动记录”。

## 7. 风险名单排序规则

用户已经确认风险名单采用混合排序，而不是只看一个维度。

排序规则定义如下：

1. 先按 `overallMastery` 从低到高排序
2. 在掌握度相同或接近的情况下，优先把 `lastActivityAt` 更久未更新的学生排前
3. `lastActivityAt` 缺失视为更高关注优先级

实现上建议使用稳定排序思路：

- 先归一化 `overallMastery`
- 再归一化活动时间为可比较值
- 对无活动记录学生给出更高风险权重

这样排序结果会更符合教师的直觉：

- 先看学得差的
- 学得同样差时，先看已经沉默更久的

## 8. 前端实现方案

本次改动集中在 `frontend/src/views/TeacherCourseOverviewView.vue` 及其测试文件。

### 8.1 数据加载

当前页面只加载课程概览接口。改造后应并行加载三类数据：

- `fetchTeacherCourseOverview`
- `fetchTeacherCourseStats`
- `fetchTeacherStudents`

推荐做法：

- 将课程建设数据错误状态与学情数据错误状态拆开
- 页面整体仍以课程概览数据为主加载门槛
- 学情区允许单独失败、单独空态

即：

- `overview` 失败：页面主内容不可展示
- `stats` 或 `students` 失败：仅学情区显示局部错误提示

### 8.2 计算属性

建议新增以下计算属性：

- `studentStats`
- `studentDistribution`
- `priorityStudents`
- `hasLearningStatusData`

其中 `priorityStudents` 负责：

- 拿到学生列表
- 做掌握度与活动时间归一化
- 按混合规则排序
- 截取前 5 名

### 8.3 跳转行为

重点关注学生项中的 `查看详情` 应复用现有学生详情路由：

`/teacher/courses/:courseCode/students/:learnerCode`

这样本页只负责发现问题，不负责承接完整分析。

## 9. UI 设计约束

本次增强应保持现有教师端视觉语言，不另起一套风格。

约束如下：

- 继续使用现有卡片边框、圆角、浅底色体系
- 学情区与课程建设区视觉权重接近，但学情区位置优先
- 风险名单采用轻量列表卡，而不是大表格
- 移动端下列表应自然折叠成单列

为了避免页面显得更“空”，建议：

- 把学情模块做成左右两栏
- 左侧为班级态势
- 右侧为重点关注学生

在窄屏下再降为单列布局。

## 10. 空态与错误态

### 10.1 学情为空

当课程暂无学生数据时：

- 班级态势显示 0 值
- 风险名单显示“暂无学生学情数据”

### 10.2 学情接口失败

当 `stats` 或 `students` 任一失败时：

- 学情区显示局部错误文案
- 原有课程基础信息、覆盖信息、章节分布仍正常显示

推荐文案：

- `未能读取班级学情，请稍后重试。`

### 10.3 学生最近活动为空

单个学生缺少活动记录时：

- 不显示空白时间
- 显示 `暂无最近活动记录`

## 11. 测试设计

测试重点放在页面行为，而不是样式细节。

至少补充以下两类测试：

### 11.1 正常渲染

验证内容：

- 同时请求 `overview`、`stats`、`students`
- 页面显示学生总数、平均掌握度、掌握度分层
- 页面显示按规则排序后的重点关注学生
- 列表项可跳转到学生详情页

### 11.2 局部失败隔离

验证内容：

- `overview` 成功，但 `stats` 或 `students` 失败
- 页面仍保留课程建设模块
- 学情区显示局部错误提示，而不是整页失败

如有余量，可补：

- 学生为空时的空态
- `lastActivityAt` 缺失时的排序与文案
