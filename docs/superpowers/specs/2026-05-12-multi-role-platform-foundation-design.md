# 多角色平台骨架设计方案（第二阶段）

日期：2026-05-12

## 1. 设计背景

当前项目已经完成学生侧第一阶段增强，包括：

1. 按掌握度分层的资源推荐与数量控制
2. 学习者画像图表化展示
3. 学习反馈后的练习检验入口流程

这些改动增强了学生端闭环，但项目整体仍然停留在“单学生演示系统”状态。当前系统大量逻辑默认围绕 `demo-learner` 展开，缺少统一账号、角色分流和平台工作台入口，导致教师与管理员职责只能停留在概念层，无法形成真实的平台结构。

因此，第二阶段的核心目标不是立即完成完整后台，而是完成一个**可登录、可分角色、可进入最小工作台、并能继续复用现有学生学习主线**的平台骨架。

## 2. 总体设计定位

第二阶段将系统定位为：

**以统一账号与角色体系为入口，学生继续使用现有学习导航系统，教师与管理员进入最小工作台的教学支持平台。**

本阶段不追求完整教学后台，而是先完成：

1. 统一账号身份层
2. 登录与会话恢复能力
3. 基于角色的页面分流
4. 学生 / 教师 / 管理员三角色最小入口
5. 学生账号与现有 learner 业务链路的映射

这个定位保证两点：

1. 不推翻当前学生端主线
2. 为教师端与管理员端后续扩展提供结构基础

## 3. 多角色方案比较与推荐路线

### 3.1 方案 A：演示壳优先

做法：

1. 新增角色登录页
2. 登录后按角色跳转
3. 教师与管理员只显示占位页

优点：

1. 改动最少
2. 上线速度最快

缺点：

1. 平台感很弱
2. 容易在答辩时被追问“登录后具体能做什么”

### 3.2 方案 B：骨架可用型

做法：

1. 新增统一 `users` 身份层
2. 保留 `learners` 作为学生学习实体
3. 支持账号密码登录与演示快捷入口
4. 学生进入现有学习系统
5. 教师进入最小工作台：课程列表、课程概览、内容维护入口
6. 管理员进入最小工作台：用户列表、课程列表、绑定关系概览

优点：

1. 平台结构完整
2. 工作量可控
3. 答辩时既能说明系统真实性，也能保证主线不过度发散

缺点：

1. 需要新增身份层、会话层和角色路由
2. 需要重新组织前端入口和部分后端接口

### 3.3 方案 C：后台实管型

做法：

1. 新增统一账号与角色体系
2. 直接进入教师端 / 管理员端的真实增删改查流程
3. 同步推进课程、知识点、用户等完整后台能力

优点：

1. 系统完整度更高
2. 为后续真实扩展打基础

缺点：

1. 范围膨胀明显
2. 容易把毕设主线从个性化学习导航带偏成教务后台开发

### 3.4 推荐路线

推荐采用**方案 B：骨架可用型**。

原因：

1. 当前系统学生端主线已经较完整，不应推翻重做。
2. 平台化需要真实入口和最小工作台，而不仅仅是角色占位页。
3. 教师与管理员先做查看型入口，最适合当前毕设节奏。

## 4. 统一用户与角色骨架设计

### 4.1 核心原则

1. 登录身份与学习画像分层
2. 学生、教师、管理员共享统一账号体系
3. `learners` 继续保留为学生学习业务实体
4. 新增身份层不能破坏当前学生端已有业务链路

### 4.2 为什么不能直接改造 learners

当前 `learners` 表主要承载：

1. 目标课程
2. 掌握度
3. 学习反馈
4. 学习画像与学习轨迹

这些语义都属于“学生学习实体”，而不是“平台登录身份”。

如果把教师和管理员也放进 `learners`，会出现以下问题：

1. 教师和管理员没有掌握度和学习路径语义
2. 角色判断会和学习业务耦合在一起
3. 数据表语义被破坏，后续扩展困难

因此必须拆成两层：

1. `users`：统一平台账号身份层
2. `learners`：学生学习业务层

### 4.3 建议新增数据实体

建议新增以下表结构：

#### users

统一平台账号身份。

建议字段：

1. `id`
2. `username`
3. `password_hash`
4. `display_name`
5. `status`
6. `created_at`
7. `updated_at`

#### user_roles

用户与角色关系表。

建议字段：

1. `id`
2. `user_id`
3. `role_code`

角色值先固定为：

1. `student`
2. `teacher`
3. `admin`

#### user_learner_links

学生账号与 learner 的映射关系。

建议字段：

1. `id`
2. `user_id`
3. `learner_id`

作用是：

1. 学生账号登录后
2. 系统知道对应哪一个 learner
3. 再复用现有学习路径、画像、反馈等业务链路

#### teacher_course_assignments

教师与课程归属关系。

建议字段：

1. `id`
2. `user_id`
3. `course_id`

### 4.4 角色关系建议

#### 学生账号

1. 必须绑定一个 learner
2. 登录后进入学生学习系统

#### 教师账号

1. 不绑定 learner
2. 可绑定一个或多个课程
3. 登录后进入教师工作台

#### 管理员账号

1. 不绑定 learner
2. 不参与学习画像和路径业务
3. 登录后进入管理员工作台

### 4.5 与当前系统的过渡接法

当前系统大量请求直接围绕 `learnerCode` 展开，因此第二阶段第一轮不建议强行重写所有学习接口。

建议过渡策略：

1. 用户登录
2. 若角色为学生
3. 后端通过当前用户找到绑定的 learner
4. 前端继续复用现有 learnerCode 学习链路

这样做的好处：

1. 学生端算法与业务接口可以少改
2. 平台身份层可先稳定落地
3. 后续再逐步从显式 `learnerCode` 过渡到基于登录态解析 learner

## 5. 登录方式与角色入口设计

### 5.1 设计目标

1. 支持账号密码登录
2. 支持演示快捷入口
3. 登录成功后按角色进入不同首页
4. 不引入过重的认证体系

### 5.2 登录页结构

登录页建议包含：

1. 用户名输入框
2. 密码输入框
3. 登录按钮
4. 演示快捷入口区

### 5.3 为什么保留演示快捷入口

毕设答辩时通常需要快速切换角色、降低现场输入风险，因此建议同时保留：

1. 真实账号密码登录
2. 预置演示账号快捷登录

演示账号建议至少包含：

1. 学生演示账号
2. 教师演示账号
3. 管理员演示账号

推荐做法：点击快捷入口后直接填充并完成登录，保持统一登录体验。

### 5.4 角色分流逻辑

登录成功后，系统读取当前用户角色并自动跳转：

1. `student` -> 学生首页
2. `teacher` -> 教师工作台首页
3. `admin` -> 管理员工作台首页

第一轮不建议做复杂多角色切换器，但结构上可以预留：

1. `currentRoles`
2. `activeRole`

### 5.5 登录态与会话恢复

建议采用轻量会话方案：

1. 后端登录成功后返回用户基础信息、角色信息，以及学生账号的 learner 绑定信息
2. 前端持久化当前会话
3. 页面刷新时通过会话恢复接口重建当前登录态

第一轮不做：

1. 注册
2. 找回密码
3. 完整 JWT 刷新体系
4. 重型 RBAC 权限引擎

## 6. 学生 / 教师 / 管理员最小工作台设计

### 6.1 学生工作台

学生端继续复用当前系统，重新定义为“学生工作台”。

保留能力：

1. 学习路径规划
2. 资源推荐
3. 学习反馈
4. 练习检验入口
5. 学习者画像
6. 学习图谱与细化学习

学生登录后的主要变化是：

1. 页面顶部展示当前用户身份
2. learner 来源不再写死为 `demo-learner`
3. learnerCode 来自登录学生账号与 learner 的绑定关系

### 6.2 教师工作台

教师端第一轮目标不是完整编辑后台，而是形成“课程负责人与教学观察者”的最小入口。

教师首页建议包含：

1. 我的课程
2. 课程概览
3. 内容维护入口

建议最小页面：

1. `/teacher`
2. `/teacher/courses`
3. `/teacher/courses/:courseCode`

第一轮这些页面以查看和入口为主，不要求完整编辑器。

### 6.3 管理员工作台

管理员端第一轮目标是形成平台治理感，而不是复杂运维后台。

管理员首页建议包含：

1. 用户概览
2. 课程概览
3. 平台管理入口

建议最小页面：

1. `/admin`
2. `/admin/users`
3. `/admin/courses`
4. `/admin/assignments`

### 6.4 为什么采用最小工作台

这种划分的优势：

1. 学生端主线不被冲掉
2. 教师与管理员登录后不是空白页
3. 平台感已经成立，但工作量仍然可控

## 7. 后端接口与数据库演进设计

### 7.1 数据库演进目标

当前数据库核心仍为课程、知识点、依赖、learners、掌握度、反馈和资源记录。第二阶段不重做这些表，而是在其上增加身份层。

### 7.2 建议新增后端接口

#### 认证与会话接口

1. `POST /api/auth/login`
2. `POST /api/auth/logout`
3. `GET /api/auth/session`

返回建议包括：

1. user 基础信息
2. role 列表
3. 若为学生，返回绑定 learner 信息

#### 教师工作台接口

1. `GET /api/teacher/courses`
2. `GET /api/teacher/courses/:courseCode/overview`

第一轮先做查看型接口。

#### 管理员工作台接口

1. `GET /api/admin/users`
2. `GET /api/admin/courses`
3. `GET /api/admin/course-assignments`

第一轮同样先做查看型接口。

### 7.3 学生端接口过渡策略

第二阶段第一轮不强改所有学生学习接口签名，而是：

1. 登录成功后拿到学生绑定 learnerCode
2. 学生端继续复用当前 learnerCode 请求链路
3. learnerCode 从登录态而不是硬编码来源提供

这样可以让：

1. 路径规划算法少动
2. 反馈、画像、资源推荐少动
3. 平台身份层与学生业务层先稳定接通

### 7.4 权限控制建议

第一轮后端角色限制建议：

1. `student` 只访问学生接口
2. `teacher` 访问教师工作台接口
3. `admin` 访问管理员工作台接口

前端路由层也同步做菜单与跳转守卫。

## 8. 前端路由、会话状态与页面结构设计

### 8.1 路由结构建议

前端建议分为四类路由：

#### 公共路由

1. `/login`

#### 学生端路由

1. `/`
2. `/profile`
3. `/resources`
4. `/learning-graph`
5. `/detail-learning`
6. `/practice-check`

学生端尽量复用现有路由，减少返工。

#### 教师端路由

1. `/teacher`
2. `/teacher/courses`
3. `/teacher/courses/:courseCode`

#### 管理员端路由

1. `/admin`
2. `/admin/users`
3. `/admin/courses`
4. `/admin/assignments`

### 8.2 会话状态建议

新增统一 `authStore`，至少包含：

1. `currentUser`
2. `currentRoles`
3. `activeRole`
4. `linkedLearner`
5. `isAuthenticated`

如果是学生账号，还要保存：

1. learnerCode
2. learnerName
3. targetCourseCode

### 8.3 页面初始化逻辑

应用启动时建议执行：

1. 调用 `GET /api/auth/session`
2. 若已有登录态，则恢复会话并按角色进入对应首页
3. 若无登录态，则跳转 `/login`

### 8.4 学生端接现有页面的方式

第二阶段学生端主要改动为：

1. learnerCode 从 `authStore.linkedLearner` 获取
2. 原本写死 `demo-learner` 的位置改为“登录态 learnerCode 优先，默认 demo 仅作过渡兜底”

这样可以逐步迁移，而不需要一次性重写学生端所有页面。

### 8.5 路由守卫建议

前端最小路由规则：

1. 未登录只能访问 `/login`
2. 学生不能进入教师/管理员路由域
3. 教师不能进入管理员域
4. 管理员进入管理员域

若路由不合法，则跳回对应角色首页或登录页。

## 9. 第二阶段实施分期与最小可交付范围

### 9.1 阶段 2A：身份层落地

目标：

1. 新增 `users`
2. 新增 `user_roles`
3. 新增 `user_learner_links`
4. 新增 `teacher_course_assignments`
5. 增加预置账号
6. 增加登录、退出、会话恢复接口

交付标准：

1. 可以登录
2. 可以识别角色
3. 学生账号能映射到 learner
4. 页面刷新后能恢复会话

### 9.2 阶段 2B：前端角色入口落地

目标：

1. 增加登录页
2. 增加 `authStore`
3. 增加路由守卫
4. 登录成功后按角色跳转
5. 学生端接入登录态 learner
6. 教师 / 管理员首页可进入

交付标准：

1. 三角色可以正常登录
2. 三角色可以进入各自首页
3. 学生登录后仍可使用现有学习主链路
4. 不同角色不能乱入其他角色路由域

### 9.3 阶段 2C：最小工作台落地

目标：

1. 教师课程列表页
2. 教师课程概览页
3. 管理员用户列表页
4. 管理员课程列表页
5. 管理员教师绑定概览页

交付标准：

1. 教师登录后不是空白页
2. 管理员登录后不是空白页
3. 后台已经具备查看型平台能力
4. 为下一步内容维护与平台治理留好入口

### 9.4 第一轮必须完成的最小可交付范围

建议压缩到以下最低范围：

1. 数据库新增用户与角色相关表
2. 预置学生 / 教师 / 管理员账号
3. 登录页
4. 登录接口与会话恢复接口
5. `authStore`
6. 路由守卫
7. 学生端切换到登录态 learner
8. 教师首页
9. 管理员首页
10. 教师课程列表 / 管理员用户列表至少各 1 页

### 9.5 第一轮明确不做的内容

1. 用户注册
2. 找回密码
3. JWT 刷新令牌体系
4. 完整 RBAC 引擎
5. 教师端完整课程编辑器
6. 管理员端完整增删改查后台
7. 真实题库管理系统
8. 多课程复杂导入流程

## 10. 结论

第二阶段最合理的落点是：

1. `users` 作为统一身份层
2. `learners` 保持学生学习实体
3. 前端新增登录与角色入口
4. 学生继续复用现有学习系统
5. 教师与管理员先拥有最小可用工作台
6. 平台感建立起来，但不陷入重型后台开发

这一方案既保住了当前毕设的核心主线，又为后续课程维护、题目维护、平台治理和多角色扩展打下了清晰且可实施的结构基础。
