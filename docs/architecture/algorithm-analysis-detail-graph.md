# 算法复杂度分析细化图谱说明

最后更新：2026-03-18

## 1. 目标

本文件用于说明“`algorithm-analysis` / 算法复杂度分析”这一课程级节点的第一版细化图谱来源、拆分依据与依赖关系设计。

当前目标仍然是围绕课程主图中的单个节点逐步下钻，让知识图谱展示层先具备“按章节继续细化”的稳定能力，再考虑是否把这些细粒度节点纳入后续规划算法。

## 3. 第一版细化节点

当前把“算法复杂度分析”拆成以下 10 个细化节点：

1. 算法定义
2. 输入与输出
3. 有穷性
4. 确定性
5. 可行性
6. 评价标准
7. 语句频度
8. 时间复杂度
9. 空间复杂度
10. 复杂度案例解析

## 4. 依赖设计原则

本细化图谱不是简单照抄教材小节，而是按“先理解算法基本定义与特性，再进入评价标准，最后进入复杂度分析”的学习顺序组织。

当前依赖主线如下：

- 算法定义 -> 输入与输出 / 有穷性 / 确定性 / 可行性
- 输入与输出 / 有穷性 / 确定性 / 可行性 -> 评价标准
- 评价标准 -> 语句频度 -> 时间复杂度 -> 复杂度案例解析
- 评价标准 -> 空间复杂度

这样处理的原因是：

- “评价标准”是从算法特性过渡到效率分析的桥梁
- “语句频度”是推导时间复杂度的基础，适合单独保留
- “复杂度案例解析”更适合作为时间复杂度的应用型后继节点
- “空间复杂度”与时间复杂度同属效率评价维度，但概念链路相对独立

## 5. 当前实现约定

- 顶层课程图谱仍然来自 SQLite 中的 `knowledge_points` 与 `knowledge_dependencies`
- 细化图谱暂通过 `backend/config/knowledge_graph_details.json` 维护
- 前端在课程主图中双击“算法复杂度分析”节点，可进入该节点的细化图谱
- 页面上方面包屑会显示当前已进入 `algorithm-analysis` scope，并支持返回课程主图
- 当前该细化节点只服务于图谱展示，不直接改变路径规划的课程级主线

## 6. 对论文与答辩的价值

这组细化节点比较适合答辩讲解，因为它能体现两个重点：

1. 系统不是只展示“章节名称”，而是可以继续拆到更细粒度知识点
2. 节点之间的边不是随意连接，而是依据教材目录与学习前置关系整理得到

在演示时，可以从课程主图中的“算法复杂度分析”节点双击进入，说明系统如何把“算法定义与特性”逐步过渡到“时间复杂度 / 空间复杂度 / 案例解析”。

## 7. 当前边界

本次只完成了第二个节点“算法复杂度分析”的细化图谱，不调整当前路径规划算法、学习记录逻辑和推荐解释生成逻辑。

也就是说：

- 当前“多层图谱下钻”仍先服务于知识图谱展示
- 路径规划仍以课程级主节点为基本单位
- 后续若继续做第三层细化，可以在 `algorithm-analysis-time-complexity` 等节点下继续挂新的 `detailScopeCode`
