## 4. 依赖设计原则

本细化图谱按“先理解栈是什么，再理解如何实现和操作，最后过渡到递归与应用”的学习顺序组织。

当前依赖主线如下：

- 定义与特点 -> 抽象数据类型
- 抽象数据类型 -> 顺序栈表示 / 链栈
- 顺序栈表示 -> 顺序栈初始化 -> 入栈操作 / 出栈与取栈顶
- 出栈与取栈顶 / 链栈 -> 递归工作栈
- 入栈操作 / 出栈与取栈顶 / 递归工作栈 -> 典型应用

这样处理的原因是：

- “栈的定义与特点”决定了后续所有操作的行为约束
- “顺序栈”和“链栈”构成两条典型实现分支
- “递归工作栈”能把栈从数据结构概念延伸到程序执行机制，适合答辩时讲解

## 5. 当前实现约定

- 顶层课程图谱仍然来自 SQLite 中的 `knowledge_points` 与 `knowledge_dependencies`
- 细化图谱暂通过 `backend/config/knowledge_graph_details.json` 维护
- 前端在课程主图中双击“栈”节点，可进入该节点的细化图谱
- 页面上方面包屑会显示当前已进入 `stack` scope，并支持返回课程主图
- 当前该细化节点只服务于图谱展示，不直接改变路径规划的课程级主线

## 6. 对论文与答辩的价值

栈这一层细化适合用来展示系统如何从“结构定义”过渡到“程序运行机制”和“典型问题求解”。

在演示时，可以从课程主图中的“栈”节点双击进入，讲清楚：

- 为什么栈满足后进先出
- 为什么顺序栈要关注栈满和栈空
- 为什么链栈能提供更灵活的空间扩展
- 为什么递归过程本质上也依赖工作栈
- 为什么数制转换、括号匹配、表达式求值都适合用栈解决
