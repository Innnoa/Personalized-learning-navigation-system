最后更新：2026-03-19

## 1. 目标

本文件用于说明“`sequence-list` / 顺序表”这一课程级节点的第一版细化图谱来源、拆分依据与依赖关系设计。

## 4. 依赖设计原则

本细化图谱按“先理解为什么顺序表可以这样存，再理解结构和空间约束，最后进入基本操作与适用判断”的学习顺序组织。

当前依赖主线如下：

- 顺序存储表示 -> 存储结构定义 / 随机访问特点
- 存储结构定义 -> 容量约束 / 初始化
- 初始化 -> 插入操作 / 删除操作 / 合并操作
- 容量约束 -> 插入操作 / 合并操作
- 随机访问特点 / 插入操作 / 删除操作 -> 适用场景

这样处理的原因是：

- “顺序存储表示”是顺序表所有实现特性的出发点
- “容量约束”是顺序表区别于抽象线性表的重要实现限制
- “适用场景”不是单独背诵的结论，而是由随机访问优势和插入删除代价共同推出来的

## 5. 当前实现约定

- 顶层课程图谱仍然来自 SQLite 中的 `knowledge_points` 与 `knowledge_dependencies`
- 细化图谱暂通过 `backend/config/knowledge_graph_details.json` 维护
- 前端在课程主图中双击“顺序表”节点，可进入该节点的细化图谱
- 页面上方面包屑会显示当前已进入 `sequence-list` scope，并支持返回课程主图
- 当前该细化节点只服务于图谱展示，不直接改变路径规划的课程级主线

## 6. 对论文与答辩的价值

顺序表这一层细化适合用来说明系统不仅能展示“概念型知识点”，也能展示“实现型知识点”的内部结构。

在演示时，可以从“线性表”主线继续进入“顺序表”，讲清楚：

- 为什么顺序表支持随机访问
- 为什么插入删除要移动元素
- 为什么容量约束会影响合并和插入
- 为什么它适合“查询多、修改少”的场景

这会让图谱展示和后续路径规划解释更贴近课程教学逻辑。

## 7. 当前边界

本次只完成了第四个节点“顺序表”的细化图谱，不调整当前路径规划算法、学习记录逻辑和推荐解释生成逻辑。

也就是说：

- 当前“多层图谱下钻”仍先服务于知识图谱展示
- 路径规划仍以课程级主节点为基本单位
- 后续若继续细化，可以把 `sequence-list-insert`、`sequence-list-delete` 等操作节点继续挂接更底层 scope
