## 1. 目标

本文件用于说明“`string` / 串”这一课程级节点的第一版细化图谱来源、拆分依据与依赖关系设计。

8. 简单模式匹配

## 4. 依赖设计原则

本细化图谱按“先理解串是什么，再理解如何表示，最后过渡到模式匹配问题”的学习顺序组织。

当前依赖主线如下：

- 定义与基本概念 -> 基本操作 -> 抽象数据类型
- 抽象数据类型 -> 定长顺序存储 / 堆分配存储 / 链式存储
- 基本操作 / 定长顺序存储 -> 模式匹配问题 -> 简单模式匹配
- 堆分配存储 / 链式存储 -> 简单模式匹配

这样处理的原因是：

- “模式匹配问题”要建立在对子串、比较等串操作的理解之上
- “简单模式匹配”是进入 KMP 之前最自然的前置内容
- 把三类存储方式保留下来，有利于后续解释串算法实现时的访问方式差异

## 5. 当前实现约定

- 顶层课程图谱仍然来自 SQLite 中的 `knowledge_points` 与 `knowledge_dependencies`
- 细化图谱暂通过 `backend/config/knowledge_graph_details.json` 维护
- 前端在课程主图中双击“串”节点，可进入该节点的细化图谱
- 页面上方面包屑会显示当前已进入 `string` scope，并支持返回课程主图
- 当前该细化节点只服务于图谱展示，不直接改变路径规划的课程级主线

## 6. 对论文与答辩的价值

串这一层细化适合用来展示系统如何从“基本字符串表示”过渡到“模式匹配问题”。

在演示时，可以从课程主图中的“串”节点双击进入，讲清楚：

- 为什么串虽然和线性表相似，但更聚焦字符处理问题
- 为什么串需要不同存储方式
- 为什么简单模式匹配是进入 KMP 的前置基础
- 为什么 KMP 可以作为后续继续下钻的独立算法节点

## 7. 当前边界

本次只完成了第八个节点“串”的细化图谱，不调整当前路径规划算法、学习记录逻辑和推荐解释生成逻辑。

也就是说：

- 当前“多层图谱下钻”仍先服务于知识图谱展示
- 路径规划仍以课程级主节点为基本单位
- 后续若继续细化，可以把 `string-naive-match` 或课程主图中的 `kmp` 继续下钻到 next 数组和失配回退等更细算法步骤
